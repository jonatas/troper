1 INTRODUÇÃO



O projeto de desenvolvimento consistirá num Sistema elaborado para gerar e executar relatórios na Web, agregado a outro Sistema já existente.

Sabe-se que projetos de software, em geral, apresentam foco em partes específicas e suas vantagens. A Leosoft conta com um bom feedback em um de seus sistemas, por trabalhar uma ferramenta para construir relatórios de maneira fácil e útil, o CoopRel. Os usuários agradam-se com a liberdade para construir / modificar relatórios para necessidades específicas. Tal software, em operação, é restrito para Desktop.

O projeto proposto é uma alternativa ao CoopRel, ampliando as possibilidades de facção de relatório, através de aplicativos Web



1.1 Objetivos

1.1.1 Gerais

O objetivo do projeto é encontrar um escopo para a construção de relatórios simples e personalizados a partir de aplicativos web.

1.1.2 Específicos

a) Construir relatórios automaticamente através de uma inteface estilo ‘passo a passo’.

b) Executar relatórios, enviando para o servidor processar, executar os filtros e apresentar os dados em HTML.

1.2 Justificativa

A maioria dos projetos Web da Leosoft utilizam o framework Ruby On Rails, que ainda não possui nenhuma ferramenta pronta para executar este tipo de trabalho.

Como solução para a empresa, os programadores e suporte dos sistemas escritos em linguagem Java e Ruby, usam iReport, que é uma ferramenta em Java para criar os relatórios para os clientes. O iReport é uma interface robusta escrita em Java, que dá a possibilidade de ajustar todos os aspectos da construção e desenvolvimento de um arquivo jasper report. Trata-se de uma biblioteca para a criação de documentos em diversos formatos (pdf, html, xsl, etc.) com leiaute personalizado a partir de um arquivo jrxml – jasper report xml. O uso do iReport para a construção de relatórios dificulta a intervenção do usuário, gerando maior demanda para empresa desenvolvedora.

Quanto à interface do software proposto, as mudanças no relatório, desde o seu formato gráfico até os dados, serão feitas independente dos desenvolvedores, tornando o usuário mais autônomo. O sistema de relatórios funcionará em sistemas já existentes na Leosoft, agregando a eles maior valor.

Dessa forma, a empresa terá uma folga significativa na construção e manutenção de relatórios, oferecendo mais um módulo interessante para o cliente.



1.2 Descrição do objeto



Entender os modelos e regras do sistema são tarefas relativamente simples em projetos Rails. Com o uso do framework, existe a possibilidade de trabalhar com as entidades e seus relacionamentos, através do 'mapeador objeto relacional' ActiveRecord. Este framework conecta objetos de negócio a tabelas de banco de dados, facilitando a criação e persistência de modelos de domínio. Dessa forma, é possível extender a fonte de dados padrão, usando as próprias definições dos modelos de dados.

O número de fonte de dados depende da maturidade do sistema alvo. Em alguns casos será necessária a elaboração de visões gerenciais e fontes de dados inteligentes conforme a demanda de fonte de dados do cliente final.



2 FUNDAMENTAÇÃO TEÓRICA



Para a fundamentação teórica deste projeto, elaboro algumas perguntas relativas ao desenvolvimento do sistema proposto. O método de desenvolvimento mais adequado é a Extreme Programming:

Extreme Programming (XP) é uma metodologia que se aproxima da realidade do ambiente de desenvolvimento de softwares. Com 8 anos de existência, vem melhorando muitas empresas de diversos tamanhos pelo mundo (...) efatiza também o trabalho em time.[1]





É uma metodologia adequada pelas suas boas práticas como: testes automatizados e princípios do manifesto ágil como mais interação entre os envolvidos que processos e ferramentas.

Testes automatizados são softwares criados para testar o próprio software antes da sua codificação. A forma como estes testes são escritos é Behaviour Development Driven:



Desenvolvimento orientado a comportamento (BDD) traz uma interação completa do desenvolvimento orientado a testes e desenvolvimento orientado a domínio, fazendo relacionamentos entre essas forças tornando o desenvolvimento de software mais evidente.[2]



2.1 Por que usar estórias do usuário ao invés de casos de uso?



Trata-se de uma questão comum. Muitos seguidores do método XP consideram 'estórias' como uma alternativa mais simples aos casos de uso, ambos ferramentas organizadoras de requisitos. São, contudo, diferentes nos seus propósitos de organização. Casos de uso organizam requisitos na forma narrativa, abrangendo os modos de relacionamento dos usuários com o sistema; mantém o foco nos alvos e sua interação com o sistemas. As estórias XP, por sua vez, fragmenta os requisitos em pequenos pedaços, permitindo assim criar uma estimativa de tempo para o desenvolvimento deste.
